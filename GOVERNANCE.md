# Governance

B.CAVE *AI 솔루션 도입 및 관리 규정 v1.0*에 따른 본 시스템의 거버넌스 명세.

## 1. 시스템 분류

| 항목 | 내용 |
|---|---|
| 시스템명 | B.CAVE Competitor Radar |
| 분류 | 사내 AI 분석 시스템 (Class B — 외부 데이터 수집형) |
| 관할 | IT팀 (소유자: IT팀장 정호철) |
| 사용 부서 | 상품기획, MD, 마케팅, 경영지원 |
| 데이터 민감도 | 자사 데이터: 내부(중), 외부 데이터: 공개(저) |
| 의사결정 권한 | 본 시스템은 의사결정 보조 역할, 최종 판단은 사람 |

## 2. AI기본법 (인공지능 기본법) 컴플라이언스

- 본 시스템은 "고위험 AI"에 해당하지 않음 (생체정보·신용평가·채용 등 미해당)
- 다만 다음 사항을 자발적으로 준수:
  - 모델 카드 작성·관리 (`docs/MODEL_CARD.md`, Phase 3에 추가)
  - LLM 출력은 "AI가 생성한 분석"임을 리포트에 명시
  - 인적 오버라이드 가능 (대시보드에서 분석 결과에 사람 코멘트 추가)
  - 로컬 LLM 사용으로 데이터 국외 이전 없음

## 3. 외부 데이터 수집 정책

### 3.1 대상 사이트
- 무신사 (musinsa.com)

### 3.2 준수 사항
- **robots.txt**: 매주 월요일 자동 점검, 변경 시 IT팀장에게 알림
- **rate limit**: 페이지당 최소 3초 대기, 동시 세션 1개
- **User-Agent**: 명시적 UA 사용 (`B.CAVE-Competitor-Radar/1.0 (internal analytics)`)
- **수집 시간대**: KST 03:00~07:00 (트래픽 영향 최소화)
- **목적 외 사용 금지**: 수집 데이터는 사내 분석만, 외부 공유·재판매 절대 금지
- **개인정보 미수집**: 리뷰 작성자 ID·닉네임 등 식별정보는 수집하지 않음 (리뷰 본문 텍스트만, 익명화하여 저장)

### 3.3 법무 검토
- 최초 운영 개시 전 법무팀 1회 검토 필수
- 무신사로부터 중단 요청 수신 시 즉시 수집 정지
- 분기 1회 운영 현황을 법무팀에 공유

### 3.4 비상 정지 절차
1. 무신사로부터 공식 항의 / IP 차단 / 봇 차단 강화 감지 시
2. n8n 워크플로우 비활성화 (IT팀장 + 인프라파트 권한)
3. 24시간 이내 법무팀·경영진 보고
4. 재개 여부는 법무팀 협의 후 결정

## 4. 데이터 접근 권한

### 4.1 워커 (Python)
- Supabase: service_role key (전체 권한)
- Snowflake: 읽기 전용 서비스 계정 (`SVC_COMPETITOR_RADAR_READER`)
- 저장 위치: 워커 컨테이너의 `.env` (Docker secret 권장)

### 4.2 뷰어 (Vercel)
- Supabase: anon key + RLS로 읽기만
- Snowflake: 직접 접근 불가 (워커가 Supabase에 미리 적재한 것만 봄)

### 4.3 사용자 (사내)
- 1차: 상품기획팀·MD팀 (Slack 채널 구독)
- 2차: 마케팅·경영지원 (Notion 페이지 열람)
- 관리자: IT팀장 + DT파트 (대시보드 운영)

### 4.4 외부 공유 금지
- 본 시스템 출력은 외부 보고·SNS·외부 컨설팅에 인용 금지
- 외부 활용 필요 시 IT팀장 사전 승인

## 5. 모델 거버넌스

- **모델 등록**: Qwen 2.5 14B Instruct (Alibaba, Apache 2.0 라이선스)
- **버전 관리**: 모델 교체 시 `agent_analyses.model_version`에 기록, 1주일 A/B 비교 후 전환
- **프롬프트 버전**: `agent_analyses.prompt_version`에 기록, Git 태그와 매핑
- **편향·환각 점검**: 월 1회 무작위 20건 샘플링하여 IT팀장이 검수
- **사용 금지 영역**:
  - 핵심 ERP 데이터에 대한 agentic 쓰기 권한 (AX 위원회 결정 사항)
  - 인사·평가·채용 의사결정

## 6. 감사·로깅

- 모든 스크래핑 요청은 `worker/logs/scrape.log`에 기록 (URL, 응답시간, 상태코드)
- 모든 LLM 호출은 `agent_analyses` 테이블에 기록 (프롬프트 hash, 토큰, latency)
- 로그 보관: 90일
- 분기 1회 IT팀장이 감사 리포트 작성, AX 위원회 정기 보고

## 7. 인시던트 대응

| 시나리오 | 대응 |
|---|---|
| 봇 차단 (Playwright 실패율 >50%) | 워크플로우 중단, 인프라파트 알림 |
| 무신사 ToS 변경 감지 | 법무 검토 후 운영 재개 |
| LLM 환각 (명백한 거짓 분석 발견) | 해당 분석 무효 처리, 프롬프트 수정 |
| 자사 데이터 유출 의심 | 즉시 시스템 정지, IT팀장 + 보안 담당 비상 소집 |
| Supabase 장애 | 큐잉 모드 (로컬 디스크에 저장 후 복구 시 재적재) |
